问题 1.  测试 OPEN CHANNEL ，发生 fail。


[分析结论]    


由于OPEN CHANNEL 命令中的 bearer type 为 default bearer (即 03) 时，STK AP 这边没有处理，导致open channel 失败。


[LOG分析]    


从log 中看到 OPEN CHANNEL 失败的原因是 OPEN CHANNEL 命令中包含的 bearer type 是 default bearer(即03)，STK AP层解析时没有对 default bearer 的解析处理，导致手机回复了 command data not understood by ME 的TR 给卡。

01-01 01:15:21.588672   908  1230 D CAT     : CommandParamsFactory: enter: process OpenChannel


//不支持OPEN CHANNEL 命令中包含的bearer type：


01-01 01:15:21.588791   908  1230 D CAT     : CAT: retrieveBearerDesc: un-understood bearer type


01-01 01:15:21.588870   908  1230 D CAT     : CommandParamsFactory: throw ResultException: CMD_DATA_NOT_UNDERSTOOD


01-01 01:15:21.590592   908   908 D CAT     : [BIP]: SS-handleMessage: msg: 10


01-01 01:15:21.590837   908   908 D CAT     : CatService: TERMINAL RESPONSE: 810301400382028281830132


01-01 01:15:21.595544   490   501 D AT      : AT> AT+STKTR="810301400382028281830132"


[解决方法]   


在STK AP侧添加对 bearer type为 default bearer的处理。


基本所有版本都已有对应的 patch。具体的修改可以查看对应的 patch。


 


 


问题2. 通过BIP更新卡中数据失败(如 STK 菜单或者SPN等)。 


[分析结论]    


当SET UP EVENT LIST 命令在 RILD 和RILJ连接前上报时，由于没有在 RILD 和RILJ 缓存，可能会被 RILD 或 RILJ 丢弃，导致BIP测试中卡无法正常跟网络服务器端进行数据交互。


 


[LOG分析]

06-27 11:53:23.980   932   932 D CAT     : CatService: call getInstance 2


06-27 11:53:24.018   932   932 D CAT     : CatService: CatService: is running


06-27 11:53:24.018   932   932 D CAT     : CatService: [CachedStk notify RIL to send cached command


//之后仍是如 1 一样，搜索 “event list” 关键字，当有event list null 的log 出现，则表示 RILJ 没有将收到的 SET UP EVENT LIST命令 上报。


由于 UNSOL_STK_EVENT_NOTIFY 从RILD传到RILJ后，只有当 mCatEventRegistrant != null 时才会通过 registrant 机制将它发给 CatService。而由于 RILJ收到的时间是11:53:18.851， CatService 初始化的时间是11:53:23.980， CatService 初始化是在 RILJ收到 UNSOL_STK_EVENT_NOTIFY 之后，所以，按原本的设计 RILJ 是不会把 UNSOL_STK_EVENT_NOTIFY 发送给 CatService。


[解决方法]   


当RILD 和RILJ 还没有建立连接时，若RILD 收到了SIM卡上报的SET UP EVENT LIST 命令，则RILD缓存该命令。若RILJ 收到了该命令，而此时  CatService 初始化还没完成，则缓存该命令，等收到 CatService 初始化完成的广播后再发送该命令到 CatService。


基本所有版本都已有对应的 patch。具体的修改可以查看对应的 patch。


 


 


问题3.  反复测试 OPEN CHANNEL 和 CLOSE CHANNEL 发现，某次 CLOSE CHANNEL 后，OPEN CHANNEL 失败。


[分析结论]    


CLOSE CHANNEL时，STK 调用DATA 模块 的stopUsingNetworkFeature()/ stopUsingNetworkFeatureGemini() 接口去断开PDP连接后，并没有等待DATA模块返回结果就直接回复了执行成功的TR给卡。


之后，有些卡在间隔很短的时间内立马上报了 OPEN CHANNEL 命令，导致当BIP调用DATA的startUsingNetworkFeature() / startUsingNetworkFeatureGemini() 接口去建SUPL类型的PDP连接时， STK先收到了DATA模块返回的 DISCONNECTED 的网络状态，以为当前网络是断开的，最终导致 open channel fail 且 手机回复了BIP ERROR 的Terminal Response 给卡。

问题5.  4G手机测试 OPEN CHANNEL，发生 fail。相比之下， 3G 手机测试OK。


[分析结论]    


从系统属性 persist.radio.ia-apn 中获取到DATA 缓存的开机默认建起的那一路 PDP使用的 iaApn的值为空，导致 open channel fail。


iaApn为空的原因：开机 ps attach的apn(即预置的 apn)带有密码，因此不会被ia-cache 缓存(即不会存入 persist.radio.ia-apn 中)。

修改思路： 


(1). DATA 在 RIL层新增一个 property： persist.radio.ia-bip-apn 。 


(2). 开机PS attach的apn 不管是否带有 password，都用persist.radio.ia-bip-apn 保存起来。 


(3). BIP 不再从persist.radio.ia-apn 中取开机PS attach的apn名字，而改为取persist.radio.ia-bip-apn 中保存的 apn名字。然后和BIP的apn名字比较。如果一样，则允许 reuse apn。 


KK2.MP13.TC9SP branch 上有对应的 patch: ALPS01963426。


 


 


问题6.  OPEN CHANNEL 测试，有时会 fail。


[分析结论]    


当OPEN CHANNEL命令中不包含apn时，BIP自行设定了一个TestGp.rs 的 apn，并将这个apn以supl类型存入到DATA 的数据库中。这种情况下，如果运营商BIP测试时，要使用的 apn 根本不是这个设定的 apn，会导致 open channel fail 或者 即使open channel成功，但 向 server端send data 后server端没有任何 response的问题。
